# 业务领域知识 · CONTEXT

> 本文件是 AI 与新成员理解本项目业务口径的单一入口。术语精确定义见 `specs/_glossary.md`;待定口径见 `specs/open-questions.md`。

## 1. 我们是谁、在做什么

环保管家服务公司(云南),每月服务约 100 家企业。核心工作流:ESO(环保服务官)现场排查 → ETO(环保技术官)审核 → 整改闭环 → 每月《环保管家健康体检报告》交付企业。EcoCheck 小程序承载这个工作流,并每天蒸馏现场经验(现场蒸馏 v2,三个闸口:现场/审核/闭环)。

本项目是把这些资产连成「环保现场执行图谱」:法律法规 × 技术规范 × 现场经验互相赋能。

## 2. 政府合作背景(关键上下文)

- 合作由**生态环境监控中心主任主动提出**;他是 AI 技术行家/发烧友,演示可以直接上架构,不用降维。
- 分工共识:**他们出法典文本 + 案例 + 权威解释;我们出真实现场经验 + 关联图谱**。
- 他们正建"法典知识库"(生态环境法典编纂背景下,含历史单行法继承关系)——我方图谱的法律层挂 `lineage_ref` 等他们的沿革数据,法典生效后引用自动迁移。
- 他们给基层执法大队做的工具"有法律依据但脱离真实企业场景"——我方图谱正好是那一半。
- 软著策略:第一创作人让给监控中心,我方第二——**换"与主管部门统一口径"**:我方问题分类法、报告口径成为带官方背书的行业口径。
- 对方进度慢 = 窗口在我方;第一个摆出拼接样板的人定义合作形态。
- **赌注不押在主任个人或政府背书上**:护城河是每天的蒸馏流,背书是放大器。

## 4. 业务核心概念

- **13 个环保维度**:环评手续 / 排污许可 / 竣工验收 / 废水 / 废气 / 噪声 / 一般固废 / 危废 / 应急 / 台账 / 自行监测 / 在线监测 / 环保设施运行 / 现场环境管理(EcoCheck 模板一级分类,图谱 S01-S13+ 映射见 eco-kb score13_mapping)。
- **问题类型(issue_type)**:标准化的现场问题分类(canonical + aliases 归一),例:危废标签不规范。是图谱的"焊接点"——唯一同时连法条、证据、整改、报告、统计的节点。
- **现场蒸馏**:ESO/ETO 在工作流三个闸口结构化记录判断;**delta(ESO 初判 vs ETO 终判)是一等专家经验**,不是日志噪音。
- **踩雷点(pitfall)**:法规条款被企业普遍误解/与实际场景脱节的知识点。大厂和政府都生产不出的纯现场资产,执法培训的弹药。
- **执行卡(card)**:《法条↔现场执行卡》,法条→现场表现→证据→整改→报告表达→踩雷点的图切片渲染。
- **缺边检测**:法条无现场覆盖=合规盲区;现场问题无法条依据=管理经验(**禁止 AI 把管理建议包装成法律要求**);踩雷密度=高频误解条款。
- **置信靠挣**:整改验收通过/驳回回写边 confidence,图是学习系统不是静态编纂。

## 5. 行业判断规则(影响 AI 输出的口径)

1. 法规知识库不可用时,不得引用具体法律条文(降级表达)。
2. 结论必须可追溯:问题 → 证据 → 法条 → 整改,每跳有 source_ref。
3. "管理经验类问题"与"法律要求类问题"在报告中必须区分表述,前者不得写"违反/不符合 XX 法"。
4. 法条引用必须带条款号,全文从 RAG 取,不凭记忆生成。
5. 重复出现(年内 >3 次)的同类问题标记为重复性管理问题。
